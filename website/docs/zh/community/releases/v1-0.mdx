# Rsbuild 1.0 发布

我们很高兴地宣布 Rsbuild 1.0 已经正式发布！

## 为什么是 Rsbuild

长期以来，使用 webpack 的开发者饱受两个问题的困扰：**构建慢和配置复杂**。

通过使用 Rust 来重写 webpack，[Rspack](https://github.com/web-infra-dev/rspack) 很好地解决了构建慢的问题，但为了与 webpack 生态保持兼容，Rspack 仍然保留了 webpack 的配置和 API，这意味着 Rspack 依然存在一定的复杂度和学习成本。

> 如果你还不了解 Rspack，请参考 [Rspack 介绍](https://rspack.dev/zh/guide/start/introduction)。

### 生态的发展

在早期，webpack 生态中出现了一些优秀的工具，比如 Create React App（简称 CRA）和 Vue CLI，它们对 React 和 Vue 提供开箱即用的支持，隐藏了复杂的配置细节。因此，React 和 Vue 用户通常更倾向于使用这些工具来创建应用，而不是从零开始配置 webpack。

随着生态的发展，Next.js、Nuxt 和 Remix 等全栈 web 框架变得流行；Vite 作为一个轻量化的构建工具，也受到了众多开发者的青睐。而 CRA、Vue CLI 则是逐渐停止了维护。

对于 webpack 生态，我们必须要承认一个让人遗憾的事实：webpack 生态缺少一个足够好用的构建工具，它既要像 CRA 和 Vue CLI 一样简单易用，又能与 Vite 一样具备轻量、启动快、插件化等特性。

当我们查看 webpack、CRA 和 Vue CLI 的 npm 下载量时，会发现仍然有大量项目在使用这些工具。例如，webpack 有约 2500 万的周下载量，CRA 有近 300 万的周下载量。在这些项目中，有很大一部分不需要使用 Next.js、Nuxt 等 web 框架的全栈特性；Vite 看起来是一个不错的选择，但我们在字节跳动的项目中实践后发现，从 webpack 迁移到 Vite 存在很高的迁移成本，并且需要面对迁移带来的新问题，例如开发环境与生产环境的构建产物不一致，以及大型项目在开发过程中页面刷新缓慢等。

### Rsbuild 的诞生

在一年多以前，我们意识到了上述问题，并开始尝试在 Rspack 的基础上，开发一个更现代、开发体验更好的构建工具 —— Rsbuild。

Rsbuild 是以 Rspack 为基础来实现的，我们为 Rsbuild 设计了易于使用的 API，并内置一套精心设计的构建配置，使它在充分发挥出 Rspack 性能优势的同时，也能解决配置复杂、上手成本高的问题。

在实现 Rsbuild 的过程中，我们向 CRA、Vue CLI 和 Vite 这些优秀的项目学习设计理念和最佳实践，并聚焦于两个使用场景来设计 Rsbuild：

- 作为一个轻量的构建工具：帮助开发者快速搭建 web 应用，为 CSR 应用提供开箱即用的支持，也支持通过扩展来支持 SSR、SSG 等模式。
- 作为一个共享的基础设施：为上层工具和框架提供 JavaScript API 和 plugin API，允许开发者基于 Rsbuild 来开发属于自己的工具或框架。

## 谁在使用

在 [Rspack 1.0 发布公告](https://rspack.dev/zh/blog/announcing-1-0) 中，我们介绍了 Rspack 正在取得快速增长，这其中约有一半的用户通过 Rsbuild 来使用 Rspack，其余用户则使用了 Rspack CLI 和 Rspack JavaScript API。

### 上层生态

在字节跳动内部，我们将 Rsbuild 作为各个 web 研发框架的基石，支持了不同场景的数千个项目，包括 desktop web 应用、mobile web 应用、跨平台 web 应用、文档站等。

在社区中，我们也基于 Rsbuild 开发了一些开源的工具和框架，比如静态站点生成器 [Rspress](https://github.com/web-infra-dev/rspress)，library 构建工具 [Rslib](https://github.com/web-infra-dev/rslib)，以及基于 React 的全栈框架 [Modern.js](https://github.com/web-infra-dev/modern.js)。我们也计划与 Remix 等优秀的社区团队一起探索集成 Rsbuild 的可能性。
